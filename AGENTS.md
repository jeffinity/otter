# AGENTS.md

## AI 代码变更流程（强制）

以下流程适用于后续所有 AI 对本仓库的代码变更，必须按顺序执行与校验。

### 1) 仓库结构与实现边界

- 本仓库是 Go CLI 工具仓库，当前主入口在 `cmd/`。
- CLI 子命令只负责参数解析、命令编排与用户输出；可复用逻辑应放入 `internal/` 或 `pkg/`。
- `internal/` 用于仓库内部实现，不对外承诺 API 稳定性。
- `pkg/` 用于可被外部复用的公共包，新增公共 API 时应保持命名清晰、职责单一，并补充必要测试。
- 新增功能优先沿用现有 Cobra 命令风格、`pkg/logx` 日志风格、`pkg/tuix` 输出风格。

### 2) Proto 与生成代码规范

- 仓库内已有 proto 文件位于 `pkg/` 与 `third_party/` 相关目录。
- 修改 `.proto` 文件后，必须同步执行对应生成流程；优先使用仓库已有 Taskfile 命令：
  - 配置或 proto 生成：`task conf -- <target>`
  - 其他生成流程：按 `Taskfile.yml` 中已有任务执行
- 生成文件必须与 proto 修改一并提交，不能只改 proto 不提交生成结果。
- 不手工编辑 `*.pb.go`、`*_grpc.pb.go`、`*_http.pb.go` 等生成文件；应通过生成命令更新。

### 3) Wire 与依赖注入

- 若后续新增 app 或引入 wire 依赖注入，修改 provider/wire 相关文件后必须执行：

```bash
task wire -- <target>
```

- 若当前改动不涉及 wire，不要新增无必要的 wire 结构。

### 4) 代码质量步骤（先后顺序固定）

代码修改完成后，必须按顺序执行：

```bash
task format
task lint
```

如果任务范围较小，也至少执行：

```bash
go test ./...
```

当默认 Go build cache 因系统权限不可写导致失败时，使用仓库内 cache 重试：

```bash
GOCACHE=$(pwd)/.build/go-cache go test ./...
```

不得因为首次测试受到本机 cache 权限影响就跳过验证。

### 5) 构建命令

本地或目标平台构建优先使用 Taskfile：

```bash
task build -- <target>
task build-linux-amd64 -- <target>
task build-linux-arm64 -- <target>
```

如果当前改动只影响普通 Go 包，可用以下命令做快速验证：

```bash
go test ./...
```

### 6) 文件规模与命名约束（强制）

- 功能方法名或变量名不应超过 5 个单词，避免单个名称过长；单测中的辅助变量和测试方法可适当放宽。
- 业务代码文件应保持小而清晰，单个 `.go` 文件原则上不得超过 `800` 行。
- 若文件接近或超过该阈值，必须按功能拆分到同目录、同 package 的多个文件。
- 新增代码优先保持函数短小，避免把命令参数解析、业务逻辑、系统调用、输出渲染混在同一个大函数里。

### 7) 参数校验分层约束（强制）

- CLI 参数合法性、必填项、互斥 flag、格式与范围校验，应优先在命令层完成。
- 领域层负责业务规则与流程编排，不重复做 Cobra 已完成的参数校验。
- 数据访问、系统调用、文件系统封装层只保留最小必要的防御性检查，例如空路径、越权路径、nil 依赖等。
- 若同一项参数校验在命令层、领域层、数据层重复出现，应收敛到最靠近输入边界的一层。

### 8) 测试要求

- 新增命令必须至少覆盖：
  - 命令存在性
  - aliases
  - hidden 状态
  - flags 与默认值
  - 关键参数校验
- 新增业务逻辑必须覆盖正常路径和主要错误路径。
- 涉及文件系统、系统命令、网络或平台差异时，应通过接口注入或 fake 实现测试，不要让单元测试依赖真实机器状态。
- 涉及 Linux-only 能力时，必须保证非 Linux 平台可以编译，并在运行相关命令时返回明确错误。

### 9) 命令文档同步约束（强制）

- README 只维护仓库总体说明、安装构建方式、顶层子命令的简要介绍和文档入口。
- `version` 这类简单子命令可直接在 README 中说明。
- `new`、`service` 这类复杂子命令必须在 `doc/` 目录下维护单独的完整说明文档：
  - `otter new` 对应 `doc/new.md`
  - `otter service` 对应 `doc/service.md`
  - `otter completion` 与 `otter config-completion` 对应 `doc/completion.md`
- 任何涉及顶层子命令、子命令别名、参数、flag、默认行为、示例、平台限制或错误语义的代码变更，都必须同步更新对应 `doc/<子命令>.md`。
- 若新增顶层子命令，必须同时：
  - 在 README 的子命令列表中增加简要说明和链接。
  - 新增 `doc/<子命令>.md`，除非该命令像 `version` 一样足够简单且已在 README 中完整说明。
  - 在命令测试中覆盖命令存在性、aliases、flags 和关键参数校验。
- 评审时若发现命令用法变更没有同步文档，视为不符合规范，必须补齐文档后再交付。

### 10) 设计文档同步约束（强制）

- `design/` 目录用于维护核心实现设计，优先描述架构边界、流程、关键数据结构、系统副作用、错误语义与测试策略，不替代 `doc/` 用户使用文档。
- 了解或修改复杂子命令实现前，应优先按需查看对应设计文档：
  - `otter new` / `internal/newapp` 对应 `design/newapp.md`
  - `otter service` / `internal/servicecmd` 等服务管理实现对应 `design/service.md`
  - `otter completion` / `otter config-completion` 对应 `design/completion.md`
- 任何涉及复杂子命令核心实现、流程编排、系统副作用、持久化路径、依赖注入、错误语义或测试策略的代码变更，都必须同步更新对应 `design/<子命令>.md`。
- 任何新增复杂顶层子命令，必须同步新增 `design/<子命令>.md`；若命令足够简单且无独立核心实现，可在 README 或 `doc/` 中说明无需独立设计文档的理由。
- 评审时若发现核心实现变更没有同步设计文档，视为不符合规范，必须补齐设计文档后再交付。

### 11) 平台与副作用约束

- 不要在测试中直接写入 `/etc`、`/usr/lib/systemd`、`/var/run`、`/data` 等系统目录。
- 需要测试路径或持久化行为时，使用临时目录或可注入的 filesystem provider。
- 不要在单元测试中真实执行 `systemctl`、`docker`、`ssh`、`psql`、`clickhouse` 等外部系统命令。
- 如必须验证外部命令参数，使用 fake executable、mock runner 或接口断言。

### 12) 变更卫生

- 不要修改与当前任务无关的文件。
- 工作区可能已有用户修改，不能回退或覆盖不是本次任务产生的改动。
- 手写代码改动使用 `apply_patch`。
- 格式化和生成命令只在确有必要时运行；若会改动大量无关文件，应停止并说明原因。
- 最终回复必须说明改动文件、验证命令，以及任何未完成或未验证的事项。
